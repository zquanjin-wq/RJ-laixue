## 1. 背景与定位

### 1.1 背景

PC 端 RJ-laixue（laixue.work）已经实现了多角色 AI 互动学习空间。但实际使用中发现：

- **通勤场景无法覆盖** — 学员出差、通勤路上没有 PC
- **培训部门反馈"学员打开率低"** — 学员报名课程后很少主动进 PC 学习
- **PC 端功能太重** — 拖拽、白板、spotlight 等交互在 6.7 寸屏幕上体验糟糕

### 1.2 移动端定位

> **手机端是 PC 端的"阉割版"——只保留通勤场景下能用的核心：文字稿 + 语音播放 + AI 问答。**

| 维度 | PC 端 | 移动端（学员阉割版） |
|---|---|---|
| 内容形态 | 幻灯片 + 文字 + 拖拽 + 白板 | **只有文字稿 + 语音**（播客/听书形态） |
| AI 角色 | 讲师 + 助教 + 同学（多角色） | **讲师 + 伴学**（讲师讲 PPT、伴学只在 AI 问答时发言） |
| 互动方式 | 点击/拖拽/spotlight | **听 + 文字阅读 + 提问** |
| 网络 | 在线（IndexedDB 缓存） | **仅在线**（依赖 4G/WiFi） |
| 学员入口 | /classroom/[id]（与创作共用） | **/m/[id]**（独立移动端路由） |
| 提问次数限制 | 无 | **每门课 5 次**（试点期间） |

### 1.3 入口与账号

- **入口**：用户从飞书收到链接 → 直接打开（不需要单独 APP）
- **账号**：复用现有 Supabase Auth，学员用同一套邮箱 + 初始密码登录
- **跨端**：移动端登录后状态与 PC 端同步（账号、已发布课件列表）
- **设计原则**：不在学员手机上"装东西"——发链接即用、用完即走

## 2. 核心场景（用户故事）

### 2.1 故事 1：通勤听课件

```
早上 8:30，张三在地铁上收到飞书消息："新员工入职培训已发布"
   ↓
点击链接，浏览器打开
   ↓
未登录 → 输入邮箱 + 初始密码 → 进入
   ↓
看到课程列表（仅显示已发布课件）
   ↓
点开《结构化思考》
   ↓
界面：顶部标题 + 当前章节文字稿（自动滚动）+ 底部播放控件 + 右上角 AI 问答入口
   ↓
点 ▶️ → AI 讲师开始朗读文字稿，进度条推进
   ↓
中途塞车 → 按 ⏸️ → 锁屏 → 到公司点 ▶️ 继续
   ↓
完成一章 → 自动加载下一章
```

### 2.2 故事 2：边听边提问

```
AI 讲师讲到"OBER 法则的 E 是 Easy（容易）"
   ↓
学员点右下角 💬 → 弹出半屏提问框（不离开播放页）
   ↓
输入："为什么 E 是 Easy 而不是 Easy to answer?"
   ↓
发送
   ↓
【此时 AI 讲师暂停】 → AI 讲师回复（语音 + 文字）
   ↓
AI 伴学接着补充（"我理解你的疑问，实际上 E 是指问题对学员来说要容易理解..."）
   ↓
对话流式展示，可向上滑动查看完整历史
   ↓
关闭弹窗 → AI 讲师从暂停处继续
   ↓
（剩余提问次数 -1，UI 显示"还剩 4 次"）
```

### 2.3 故事 3：学习进度跟踪

```
学员在某章中途暂停 → 关掉浏览器
   ↓
下次打开同一个链接 → 自动恢复到上次的章节 + 播放位置
   ↓
底部显示"已完成 3/8 章 · 共 24 分钟"
```

### 4.4 设计原则

- **一屏一任务** — 不堆叠功能，避免移动端过载
- **AI 伴学只在问答时出现** — 播放时只有讲师单声道，**避免两个人同时说话**
- **章节切换用按钮** — 不用手势，博客形态（用户偏好）
- **省电** — 锁屏暂停语音 / 后台自动停止
- **无障碍** — 大字号支持（系统设置自动适配）

## 5. 技术要点

### 5.1 复用 vs 重写

| 模块 | 复用 PC 端 | 移动端新建 |
|---|---|---|
| Supabase Auth | ✅ 直接复用 | |
| Supabase Data（courses / students） | ✅ 直接读 | |
| 语音生成 API | ✅ 调用同一 service_role | |
| /api/chat 后端 | ✅ AI 问答复用 | |
| Tailwind 组件库 | ✅ | |
| OPENMAIC stage store | | ❌ 不需要（无白板） |
| LangGraph 多 agent | | ❌ 简化：只讲师 + 伴学 |
| 拖拽归类 / 互动页面 | | ❌ 阉割版完全不展示 |

### 5.2 新增代码

```
app/m/                              ← 移动端路由（独立）
├── layout.tsx                       ← 移动端 layout（紧凑）
├── page.tsx                          ← /m 课程列表
└── [id]/
    ├── page.tsx                      ← /m/[id] 播放页
    └── components/
        ├── TextScript.tsx            ← 文字稿（自动滚动）
        ├── AudioPlayer.tsx           ← 语音播放器
        ├── AIQuestionDialog.tsx      ← AI 提问弹窗
        └── ProgressBar.tsx           ← 进度条

lib/mobile/                          ← 移动端专用工具
├── audio-cache.ts                    ← 语音缓存
├── progress.ts                       ← 学习进度（localStorage）
└── question-limit.ts                 ← 提问次数限制
```

### 5.3 关键决策

- **路由独立 `/m/[id]`**：不与 PC 端 `/classroom/[id]` 耦合
- **不创建新的"mobile course"概念**：移动端读同一份 `public.courses` 表
- **学习进度存 localStorage**：P0 阶段不依赖服务端
- **AI 提问复用 `/api/chat`**：后端已修复 Q&A 单轮问题（commit `e88c55c`）
- **提问次数限制存服务端**：客户端 + 服务端双校验，防止篡改
- **AI 伴学只在问答流里出现**：播放流只有讲师（避免双声道干扰）
- **章节切换用按钮**：用户偏好"博客形态"，按钮最简单
- **倍速 0.75x / 1x / 1.25x / 1.5x**：4 档预设

### 5.4 互动页面的处理

> **阉割版不展示任何拖拽 / 白板 / 互动页面**——这些类型（quiz / interactive）直接**显示文字稿+语音**，跳过互动。

- 自动跳过 quiz / interactive / whiteboard 等非 slide 类型
- 只播 narration 文字 + 语音
- 学员体验：跟普通章节一样，只是不能点击互动元素

## 6. 验收标准

### 6.1 功能验收

- [ ] 学员点击飞书链接 → 浏览器打开 → 自动登录（如已登录）或弹登录页
- [ ] 课程列表显示该学员可看的所有课件
- [ ] 进入课件 → 顶部标题 + 文字稿（自动滚动）+ 章节按钮 + 倍速 + AI 提问入口
- [ ] 点 ▶️ → AI 讲师朗读文字稿，进度条推进
- [ ] 倍速切换 0.75x / 1x / 1.25x / 1.5x 实时生效
- [ ] 上一章 / 下一章按钮工作
- [ ] 点 💬 → 半屏弹窗 → 输入问题 → AI 讲师回复 + 伴学补充
- [ ] **关闭弹窗后 AI 讲师从暂停处继续**
- [ ] **播放流只有讲师单声道，没有伴学同时发言**

## 7. 风险与开放问题

| 项 | 风险 | 解决 |
|---|---|---|
| 移动端 /m 路由与 /classroom 路由代码重复 | 后期维护成本 | 共享 `lib/audio/` + `lib/chat/` 工具类 |
| MiniMax TTS 音频自动播放 | iOS Safari 限制 | 提示用户先点 ▶️ 解锁 |
| 飞书消息点击 → 微信内置浏览器 | UA 检测 | 自动检测提示"建议用浏览器打开" |
| 提问次数限制客户端可篡改 | 安全问题 | 服务端 `course_progress_events` 表双校验 |
| localStorage 清缓存丢失进度 | UX 问题 | 暂接受，P2 上报服务端 |
