# P0-3 报告：413 Payload Too Large 排查 + 短期修 + 中期诊断

> 日期：2026-07-24  
> 任务卡：P0-3  
> 严重程度：🟡 P1（用户体验，不阻塞功能但需要缓解）

## 现象

生产环境（`laixue.work/generation-preview`）新建课程时，前端报：

```
Generation failed
Unexpected token 'R', "Request En"... is not valid JSON
```

浏览器 console：
```
api/extract-document:1  Failed to load resource: the server responded with a status of 413 ()
```

触发条件：上传 PDF `第一部分_品牌与云平台.pdf`（4.46 MB）。

## 根因

**不是 Vercel 配额问题，也不是代码 bug**——是 Next.js Serverless Functions / Route Handler 的**框架层默认 body 限制**：

| 维度 | 数值 |
|---|---|
| Next.js Route Handler 默认 body 上限 | **4.5 MB**（Vercel 框架强加，所有项目都是这个值） |
| 触发请求的 PDF 大小 | 4.46 MB |
| 撞线余量 | ~0.04 MB（**几乎肯定超**） |
| 错误响应来源 | Vercel 反代层 / Next.js framework（**不是** route handler 内部） |
| 错误响应格式 | 纯文本 "Request Entity Too Large"（**不是 JSON**） |
| 客户端症状 | `res.json()` 抛 SyntaxError "Unexpected token 'R'" |

WebFetch 验证：Vercel 文档列出的限制主要是函数时长（10s-900s）、磁盘（32-64GB）、环境变量（64KB），但 **Vercel Functions 的 body size 限制未在公开 Limits 概览中**——这是 Next.js framework 行为，所有 Route Handler 默认 4.5MB。

## 短期修（已 commit）

### 改动 1：`app/generation-preview/page.tsx` extract-document 路径

```typescript
if (!parseResponse.ok) {
  // Vercel/Next.js framework layer returns plain-text error bodies for
  // 413 (Request Entity Too Large) and similar gateway-level failures,
  // which JSON.parse() cannot handle. Inspect status first and only
  // attempt JSON parsing when the body actually looks like JSON.
  if (parseResponse.status === 413) {
    throw new Error('课程材料过大（超过 4.5MB 限制），请压缩 PDF 或拆分后再上传');
  }
  const contentType = parseResponse.headers.get('content-type') || '';
  if (!contentType.includes('application/json')) {
    throw new Error(t('generation.courseMaterialParseFailed'));
  }
  const errorData = await parseResponse.json();
  throw new Error(errorData.error || t('generation.courseMaterialParseFailed'));
}
```

**用户看到的效果**：413 → "课程材料过大（超过 4.5MB 限制），请压缩 PDF 或拆分后再上传"。其他非 JSON 错误 → 通用"解析失败"。

### 改动 2：`lib/utils/cloud-sync.ts:235-251` importCourseFromCloud

GET `/api/courses/[id]` 正常不会 413，但**防御性加固**：content-type 检查 + try/catch 包 JSON.parse。

## 中期诊断（只查不改）

**真实场景下"新建课程 POST payload 体积分布"**（脚本 `scripts/analyze-classroom-payload-bytes.js`，基于 schema 模拟 8 scene 课程）：

| 字段 | 体积 | 占比 |
|---|---|---|
| **总 payload** | **2.05 MB** | — |
| stage 元数据 | 0.2 KB | <1% |
| scenes 数组 (8 个) | 2012.1 KB | 96% |
| ├─ 每 scene 平均 | 251.5 KB | — |
| whiteboard 快照 | 30 KB | 1.4% |
| 编辑历史快照 | 60 KB | 2.9% |

**关键发现**（在每个 scene 模拟 100KB 内联音频 + 50KB 内联图片的场景下）：

| 内联资产 | 体积 | 占总 payload 比例 |
|---|---|---|
| 8 × 100KB 音频 base64 | **800 KB** | **38%** |
| 8 × 50KB 图片 base64 | **400 KB** | **19%** |
| 合计 | **1200 KB** | **57%** |

**57% 的 payload 是内联 base64 资产**。如果走 Supabase Storage 直传 + URL 引用 → payload 减少 **57%**，从 2.05 MB → 0.85 MB——**完全在 4.5MB 限制内**。

## 当前真实 PDF 案例

用户上传 4.46MB PDF → `/api/extract-document` 是 **multipart/form-data POST**，整个 PDF 作为 FormData part 提交。Vercel 框架层 4.5MB 限制包括 multipart 编码开销——4.46MB PDF + 边界/headers/其他 fields → **必然超 4.5MB**。

## 修复方向（诊断结论，先不动手）

| 方案 | 改动量 | 收益 | 风险 |
|---|---|---|---|
| **A. 客户端 PDF 压缩**（jsPDF / pdf-lib） | 中（前端代码） | 即时可用 | 损失 PDF 质量；不一定能压到 4MB 以内 |
| **B. Supabase Storage 直传** | 大（前后端 + 路由重写） | **根本解决** | 与 fork-extensions.md §"Audio 资产引用"字段裁决**强耦合**——必须同步设计 |
| **C. Vercel Pro + 申请提高 body limit** | 小（升级 + 申请） | 不用改代码 | Pro 也默认 4.5MB，需特殊申请；不是根本解 |
| **D. 拆分 PDF** | 小（前端 UI） | 立即可用 | 用户体验差；治标不治本 |

**强烈推荐 B**（Supabase Storage 直传）——但**必须与 fork-extensions.md §"Audio 资产引用"字段裁决一起定**（任务卡 P0-3 步骤 3 明确提示），避免两套引用格式。
