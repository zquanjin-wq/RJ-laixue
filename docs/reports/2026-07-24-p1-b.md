## 资产守卫设计

### 双模式

```typescript
function readGuardMode(): GuardMode {
  const raw = process.env.NEXT_PUBLIC_DSL_ASSET_GUARD_MODE;
  return raw === 'error' ? 'error' : 'warn';
}
```

默认 `warn`（防呆）—— 设置为 `error` 时，资产 URL 不合规 → `valid: false`；warn 模式下 → `valid: true` 但带 `warnings`。

### 检测规则

- 值必须以 `https://` 开头
- 检测到 `data:` URI → 显式拒绝："检测到内联 base64 资产，请先上传至 Storage 后再保存（fieldPath）"

### 切换策略

- 当前 `warn`（P0-3 复发的存量 base64 课程不会被卡住）
- **资产迁移卡完成后** → Vercel Dashboard 改 `NEXT_PUBLIC_DSL_ASSET_GUARD_MODE=error` 一行 + redeploy
- 万一 error 模式上线后发现漏网 → 改回 `warn`，无需回滚代码

## runtimeOnly strip 设计

### 关键纪律

**深拷贝后剔除**——绝不原地 delete。原地改会把内存里 stage 对象的 seq/currentSceneId 直接抹掉，播放立刻出 bug。

```typescript
export function stripRuntimeOnly<T>(stage: T): T {
  // 深拷贝（structuredClone → JSON 兜底）
  const cloned = deepClone(stage) as Record<string, unknown>;
  // 按 registry RUNTIME_ONLY 清单剔除
  // 保留 CLOUD_PERSISTED 字段
  return cloned as T;
}
```

### ⚠️ 上游"自带 bug"判定撤销

第一次跑 tsc 报"上游 test 找不到 CHAT_RUNTIME_ROLES 等 export"—���以为是上游 commit bug。**实际诊断**：

| 现象 | 真实原因 |
|---|---|
| `index.ts` 似乎缺 re-export | **已经**有 `export * from './runtime.js'`（grep 确认） |
| tsc 找不到 export | RJ 根 `tsconfig.json` `paths` 没有 `@openmaic/dsl` mapping，tsc 看不到 workspace 包 |

### 修法（方案 A2，**非方案 A**）

`tsconfig.json` 加 paths mapping：

```json
"paths": {
  "@/*": ["./*"],
  "@openmaic/dsl": ["./packages/@openmaic/dsl/src/index.ts"]
}
```

**改动属于 RJ 根 tsconfig**——**不是上游包代码**。满足"上游包一行不碰"原则。

⚠️ **Phase 4 rebase 风险**：上游改包名 / 入口路径时此 mapping 失效。**症状友好**：tsc 立刻大面积"找不到模块"，不会静默腐烂。已登记到 PROJECT-STATE.md rebase 检查清单。

### 为什么不是方案 A（改 index.ts re-export）

方案 A 的诊断前提（"index.ts 缺 re-export"）**不成立**——index.ts 已经 export * from runtime.js。改 index.ts 不仅没必要，还**违反原则**（侵入上游包）。
